Summary of Migration Actions
----------------------------

·         **Integrated into Knowledge Base(KB)**: Several documents will be absorbed into docs/knowledge-base/ under the appropriatecategory:

·         _Domain Knowledge additions_: The strategy write-ups (e.g. Bollinger Bands Strategy, MACD strategy,Married Put) are merged into the KB. This enriches the domain knowledge withconcrete strategy examples, under **Signals/Technical** or **Strategy/Options** as appropriate (ensuring no duplication with existing KB indicator articles)[_[50]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/knowledge-base/02_signals/technical/overlays/bollinger_bands.md#L1-L8)[_[51]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/strategies/bollinger-bands.md#L1-L9).

·         _Engine/Architecture knowledge_: The general system monitoring doc has already been moved into KB(Execution category)[_[19]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/knowledge-base/05_execution/monitoring.md#L1-L9), so the duplicate in architecture is removed. Other architecture docsremain outside KB for now (see below).

·         _Prompt/LLM knowledge_: No standalone “prompt” section is created in KB; internal evaluations(Claude connectors, RAG details) remain in internal docs or architecture. Inthe future, key learnings (e.g. a high-level summary of RAG approach) could bedistilled into a KB article, but the existing files are kept internal due totheir technical depth.

* _Source/Reference knowledge_: The dataset management and quick-reference guides remain in the Guides section rather than KB (they are how-to documents). However, connectors quick-reference and the Phase1 API reference are moved to a new **Reference** section of the docs site[_[10]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/style-guide.md#L69-L76), since they are technical reference materials (API docs, cheat-sheets) rather than narrative knowledge articles.
* **Relocated to Dedicated Sections**:

·         _User Guides (_docs/guides/_)_: The dataset management guide and quickreference are moved here (alongside CLI Usage, etc.), since they instruct userson using data tools. All guide filenames are normalized to kebab-case[_[41]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-rename-plan.md#L60-L63). The Guides index is updated to include the new data guides (andremove the UI proposal)[_[45]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/session-exports/session-export-20251208-docs-maintenance.md#L25-L28).

·         _Reference (_docs/reference/_)_: A new **Reference** section is(re)established for technical reference docs. The Phase 1 API Reference andConnectors Quick Reference now reside here, matching the documentation styleguide structure[_[10]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/style-guide.md#L69-L76). These documents will have frontmatter indicating their scope (e.g. domain: Engine or similar) and will belinked from relevant guides or architecture pages as needed.

·         _RFCs/ADRs (_docs/rfc/ _and_ docs/decisions/_)_: Internal proposals and design decisions aremoved to dedicated sections for better organization. The UI ImprovementProposal becomes an RFC (work-in-progress proposal)[_[44]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/style-guide.md#L80-L83). The Clean Architecture Migration doc is formatted as an ADR in docs/decisions/[_[9]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/style-guide.md#L75-L80), capturing that architectural decision process outside of the mainuser docs.

·         _Internal (_docs/internal/_)_: Many purely internal or development-relateddocuments are moved here to be maintained for team reference without appearingin published documentation. This includes evaluation reports (Claude connectorseval, MCP tools eval), internal how-tos (MCP tools quick start, branchworkflow), testing docs (Phase1 testing setup, ProofBench guide, User testingguide), the strategy template, and documentation update logs. These will havefrontmatter tags like status: internal or similar to distinguish them. They will not be part of the MkDocsnavigation for end users.

* _Operations_: (Optional) We did not explicitly create a docs/operations/ section in this plan, but if needed, dataset management could alternatively live under an Operations section per style guide[_[52]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/style-guide.md#L73-L80). Currently we placed it in Guides since it’s user-facing. If the team prefers, a new Ops section can be introduced later for things like deployment procedures, runbook, etc., and that guide can be moved there.
* **Archived** (docs/archive/): Obsolete, redundant, or superseded documents are collected here:

·         _Historical planning and statusdocs_: All files under docs/planning/ (Phase1 plans from Dec 2025)are archived as they have served their purpose. The entire planning folder is effectively moved underarchive (or removed). Similarly, outdated project docs (initial scope, oldstatus/next steps) are archived[_[46]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/session-exports/session-export-20251208-docs-maintenance.md#L55-L59).

·         _Session transcripts and logs_: The docs/session-exports/ directory is movedunder docs/archive/ (preserving the chronologicalindex if needed). These 10+ session exports (now renamed in kebab-case[_[49]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-rename-plan.md#L126-L135)) will be retained only for historical record. They will be excludedfrom the active docs navigation (the Session Exports index can be dropped).

·         _Superseded technical docs_: System Capabilities Assessment (Jan 2025) is archived as its contentis outdated by Phase1 completion[_[32]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/session-exports/session-export-20251208-docs-maintenance.md#L56-L59). The documentation update report and reference generation summary arealso archived – they are logs of maintenance activity, not enduring knowledge.

* _To-Do lists and scratch docs_: The development TODO list is removed or archived, since tasks are tracked elsewhere. Any similar scratch notes are not kept in the formal docs going forward per standards[_[53]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/references/standards/file-naming-scheme.md#L16-L25)[_[54]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/references/standards/file-naming-scheme.md#L34-L41).
* **Docs Structure Adjustments**:

·         The top-level docs/index.md and sidebar will be adjustedto reflect the removals and moves. Sections for **Architecture**, **Guides**, **Reference**, **Knowledge Base** remain. **Project** and **Strategies** sections will be removed from the navigation (their content either absorbed orarchived). An **Internal** section is not exposed on the public site (it’sjust a repository folder for maintainers).

·         The **Knowledge Base** (docs/knowledge-base/) remains the centralorganized body of domain knowledge. After migration, it will be augmented witha few strategy case studies as noted. The KB index and internal links should beupdated if new files (e.g. strategy examples) are added[_[55]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/session-exports/session-export-20251208-docs-maintenance.md#L156-L164).

·         Cross-reference updates arecritical. After renaming and moving files, all hyperlinks in the docs must beadjusted to the new paths. This includes:

o   Architecture index and docslinking to one another (use new kebab-case filenames[_[37]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-rename-execution-summary.md#L32-L40)[_[16]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-rename-execution-summary.md#L44-L53) or remove links if the target moved to archive).

o   Guides index updated to new guidefilenames[_[41]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-rename-plan.md#L60-L63).

o   Any mention of moved docs (e.g.,references to planning or status docs in other narratives) should be eitherremoved or pointed to the new location if we intend readers to find archivedcontent.

o   The provided automated script canassist with updating references post-rename[_[56]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-rename-execution-summary.md#L79-L88), but additional manual fixes are needed for moved/merged documents(e.g., linking Bollinger strategy content within the KB rather than to aseparate file).

In summary, **active user-facing documentation will be concentrated inthe Knowledge Base, Architecture, Guides, and Reference sections**, with allfiles using consistent lowercase-kebab naming[_[57]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-naming-standard-compliance-report.md#L24-L32). Internal and outdated documents are cleanly separated out of the maindocs hierarchy (into Internal or Archive), improving navigation andmaintainability. These changes align the docs with the project’s documentationstandards and make it easier for readers to find authoritative, up-to-dateinformation in the appropriate place[_[6]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/references/standards/file-naming-scheme.md#L36-L44)[_[58]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/style-guide.md#L52-L60). All cross-references will be updated to ensure no broken links afterthe migration (per the rename plan’s guidance and scripts[_[59]_](https://github.com/keith-mvs/ordinis/blob/af5c281405cea8dbda5fbf2b94cafc4e55e655be/docs/docs-rename-execution-summary.md#L84-L93)). The result will be a leaner, well-organized docs/ directory that clearly distinguishesenduring knowledge base material from ephemeral or internal documentation.
